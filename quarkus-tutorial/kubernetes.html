Cloud Native Deployment

At the end of this chapter you will be able to :

  • Containerize the Quarkus Application

  • Build and run the Quarkus Application in Kubernetes

Prerequisite

The following checks ensure that each chapter exercises are done with the right environment settings.

INFO: Make sure to use separate terminal windows or tabs between commands you run to target Docker for Mac/Windows and commands for Minikube/minishift as both will point to different docker engines.

  • Minikube

  • Minishift

  • Set your local docker to use minikube docker daemon

eval $(minikube docker-env)

  • Kubernetes should be v1.12+

kubectl version

  • Set your local docker to use minishift docker daemon

eval $(minishift docker-env)

  • OpenShift CLI should be {openshift-version}

oc version

  • Make sure to be on quarkus-tutorial OpenShift project

oc project -q

If you are not on quarkus-tutorial project, then run following command to change to quarkus-tutorial project:

oc project quarkus-tutorial

Containerize Quarkus Application

  • JVM mode

  • Native mode

./mvnw -DskipTests clean package

After successful maven build, run the following command to build the container - choose the terminal of the target environment you wish to use (docker, minishift or minokube) as you want to build and publish the image in the right registry:

docker build -f src/main/docker/Dockerfile.jvm -t example/{quarkus-project-name}:1.0-SNAPSHOT .
./mvnw -DskipTests clean package -Pnative -Dnative-image.docker-build=true (1)
1 Using the -Dnative-image.docker-build=true is very important as need a linux native binary what will be containerized.
Native compilation will take few minutes to complete.

After successful maven build, run the following command to build the container - choose the terminal of the target environment you wish to use (docker, minishift or minokube) as you want to build and publish the image in the right registry:

docker build -f src/main/docker/Dockerfile.native -t example/{quarkus-project-name}:1.0-SNAPSHOT .
For sake of simplicity the container name in both native and jvm mode are named identically. In real cases where you might need both containers, you’d like to name them differently to know exactly the package they are containing.

After containerizing the application, we can verify the container by running the following commands:

docker run -it --rm -p 8080:8080 example/{quarkus-project-name}:1.0-SNAPSHOT

You can Invoke Service to see a response like hello.

Invoke Service

The simple hello REST URI can be called via browser using http://$(minikube ip)/fruit or using CLI like:

curl $(minikube ip):8080/fruit

Then you can stop the container by typing kbd:[Ctrl+C].

Deploy Quarkus Application in Kubernetes

If you have not deployed the application previously, you need to add Quarkus Kubernetes extension. This extension uses Dekorate to generate a default Kubernetes resource template.

./mvnw quarkus:add-extension -Dextensions="quarkus-kubernetes"

You need to configure group and name of the Docker container used to deploy into Kubernetes.

Add the following properties (if you haven’t already done it before) to $PROJECT_HOME/src/main/resources/application.properties:

kubernetes.group=example
quarkus.application.name={quarkus-project-name}

Now you need to run Maven goal to generate Kubernetes resource.

./mvnw package -DskipTests

You can inspect the generated file by accessing next file:

cat target/kubernetes/kubernetes.yml

Apply Kubernetes Resources

  • kubectl

  • oc

kubectl apply -f $TUTORIAL_HOME/target/kubernetes/kubernetes.yml
oc apply -f $TUTORIAL_HOME/target/kubernetes/kubernetes.yml

A bug in the Dekorate library prevents Quarkus 0.27.0 to set the proper image name (see bug report). You can check the failure by running kubectl get pods and check that the fruit one is 0/1 and fails due to an image error.

Edit the kubernetes resource to update the image via kubectl edit deployment fruits-app and search for the image property. Change the prefix from your username (or developer) to example which is the correct group under which we published our image earlier. Save and exit and the deployment should succeed.

Invoke Kubernetes Service

  • kubectl

  • oc

You need to create a NodePort to expose the application if you are in minikube. As generated resources creates a ClusterIP service, you need to patch it:

kubectl patch svc {quarkus-project-name} --type='json' -p '[{"op":"replace","path":"/spec/type","value":"NodePort"}]'
SVC_URL=$(minikube service -n quarkustutorial {quarkus-project-name} --url)

With OpenShift we can create a route to make Quarkus application publicly accessible:

oc expose service {quarkus-project-name}

Once the service is exposed we can use the following command to get the public url:

SVC_URL=$(oc get routes {quarkus-project-name} -o jsonpath='{.spec.host}')

Now you can use the SVC_URL to call the service from the browser or via cli like:

curl $SVC_URL/{svc-path}

Delete Kubernetes Resources

  • kubectl

  • oc

kubectl delete all --all
oc delete all --all
