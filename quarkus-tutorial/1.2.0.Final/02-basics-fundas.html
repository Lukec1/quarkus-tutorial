Basics and Fundamentals

At the end of this chapter you will be able to :

  • Create a Quarkus Java Application

  • Build and run the Java application in JVM mode.

  • Build and run the Java application in Native mode.

  • Build and run the Java application in Developer mode

Create Quarkus Application

Create an empty folder called quarkus-tutorial. We’ll call it our $TUTORIAL_HOME.

mkdir quarkus-tutorial && cd quarkus-tutorial
export TUTORIAL_HOME=`pwd`

Create the fruits-app folder and navigate to the folder:

mkdir -p $TUTORIAL_HOME/work && cd $TUTORIAL_HOME/work
# set the Quarkus release environment variable
export QUARKUS_VERSION=1.2.0.Final
mvn "io.quarkus:quarkus-maven-plugin:${QUARKUS_VERSION}:create" \
  -DprojectGroupId="com.example" \
  -DprojectArtifactId="fruits-app" \
  -DprojectVersion="1.0-SNAPSHOT" \
  -DclassName="FruitResource" \
  -Dpath="fruit"
cd fruits-app

Open the Project in IDE

Open or import the generated project into your IDE.

If you are using vscode you can open the project by:

code .
All further sections of this tutorial assumes the application name to be fruits-app.

INFO: If you use tutorial-tools, your host quarkus-tutorial/work contains the application code ; point your IDE to this directory.

Build and Run the Quarkus Application

  • JVM mode

  • Native mode

  • Native container-build mode

cd $TUTORIAL_HOME/work/fruits-app && \
./mvnw -DskipTests clean package

To run the application in jvm mode

java -jar target/fruits-app-1.0-SNAPSHOT-runner.jar
cd $TUTORIAL_HOME/work/fruits-app && \
./mvnw -DskipTests clean package -Pnative
Native compilation will take few minutes to complete.

To run the application in native mode

./target/fruits-app-1.0-SNAPSHOT-runner

Tests will fail on Windows as there is no graalvm support for windows, you can workaround it using -DskipTests option.

 
cd $TUTORIAL_HOME/work/fruits-app && \
./mvnw package -DskipTests -Pnative -Dquarkus.native.container-build=true
Native compilation will take few minutes to complete.

To run the application in native mode you need to build a Docker image using the native file generated in previous step:

docker build -f src/main/docker/Dockerfile.native -t example/fruits-app:1.0-SNAPSHOT . && \
docker run -it --rm -p 8080:8080 example/fruits-app:1.0-SNAPSHOT

You can Invoke Service to see a response like hello.

Testing Fruit Resource

The scaffolding process also creates a test case for the fruit endpoint.

Open the next class $PROJECT_HOME/src/test/java/com/example/FruitResourceTest.java to see how a test looks like.

When running this test, the application is started once, then all tests are executed, and finally, the application is stopped. Although it is not mandatory, by default RestAssured project is used to test Rest endpoints but it is up to you to change that.

You can run the test either in IDE or by running Maven:

./mvnw clean compile test

Run Quarkus Application in Development mode

Stop the Quarkus application that might be running in jvm or native mode.

Development mode enables hot deployment with background compilation, which means that when you modify your Java files and/or your resource files and refresh your browser, these changes will automatically take effect.

If you are not in dev mode, run the following command to start Quarkus application in development mode:

./mvnw compile quarkus:dev

Updating the code

Only if running in Development mode

You can edit the Java source(s) in your IDE. When you save and hit refresh in your browser, you will notice browser showing the new changes.

Invoke Service

The simple hello REST URI can be called from the browser using http://localhost:8080 or using cli in a new terminal window like:

curl localhost:8080/fruit

Configuring Logging

You can configure Quarkus logging by setting the following parameters to $PROJECT_HOME/src/main/resources/application.properties:

quarkus.log.console.level=INFO

Also you can configure your application logger level. Let’s add a logger and configure it.

Open the following Java class $PROJECT_HOME/src/main/java/com/example/FruitResource.java and copy the next content in its body:

package com.example;

import javax.ws.rs.GET;
import javax.ws.rs.Path;
import javax.ws.rs.Produces;
import javax.ws.rs.core.MediaType;

import org.jboss.logging.Logger;

@Path("/fruit")
public class FruitResource {

    Logger logger = Logger.getLogger(FruitResource.class);

    @GET
    @Produces(MediaType.TEXT_PLAIN)
    public String hello() {
        logger.info("Hello method is called.");
        return "hello";
    }
}

And configure INFO mode for the service to $PROJECT_HOME/src/main/resources/application.properties:

quarkus.log.category."com.example".level=INFO

Invoke Service

The simple hello REST URI can be called from the browser using http://localhost:8080 or using cli like:

curl localhost:8080/fruit

And if you inspect the console output, you see the logging message:

2019-12-11 14:00:56,219 INFO [com.exa.FruitResource] (vert.x-worker-thread-2) Hello method is called.

Configuration

Hardcoded values in your code is a no go, let’s see how to add configuration to your application. Quarkus relies on MicroProfile Config specification and the main configuration is application.properties.

Add a new property

Add the following property to $PROJECT_HOME/src/main/resources/application.properties:

greetings.message = Hello y'all!

Get Configuration Value

Open the following Java class $PROJECT_HOME/src/main/java/com/example/FruitResource.java and copy the next content in its body:

package com.example;

import javax.ws.rs.GET;
import javax.ws.rs.Path;
import javax.ws.rs.Produces;
import javax.ws.rs.core.MediaType;
import org.eclipse.microprofile.config.inject.ConfigProperty;

@Path("/fruit")
public class FruitResource {

    @ConfigProperty(name = "greetings.message")
    String message;

    @GET
    @Produces(MediaType.TEXT_PLAIN)
    public String hello() {
        return message;
    }
}

Invoke Service

The simple hello REST URI can be called from the browser using http://localhost:8080 or using cli like:

curl localhost:8080/fruit
